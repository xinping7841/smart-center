# 小米空调插座二代 + 温湿度传感器对接说明

## 结论

这两类设备不要按同一种“底层协议设备”处理。

- 蓝牙温湿度传感器建议走 `蓝牙采集器 -> 本项目 push`，或者 `蓝牙网关/Home Assistant -> MQTT -> 本项目`
- 小米空调插座二代/空调伴侣二代建议单独作为 `空调控制设备` 处理

当前项目原有环境监测只支持 `Modbus/PRSense` 读寄存器，本次已补充 `MQTT` 型环境传感器接入能力，适合接入 Zigbee 温湿度传感器。
另外已补充 `push` 型环境传感器接入能力，适合蓝牙温湿度传感器桥接接入。

空调伴侣这边，推荐顺序是：

1. 先尝试 `miio` 本地直连
2. 如果本地直连失败，再考虑接入 Home Assistant 作为桥

Home Assistant 不是前置条件。

## 推荐拓扑

方案 A：

1. 温湿度传感器加入 Zigbee 网关
2. 网关使用 Zigbee2MQTT 发布 JSON 数据到 MQTT Broker
3. 本项目订阅对应 topic
4. 数据写入 `/api/env/status`

方案 B：

1. 蓝牙温湿度传感器被 Windows/ESP32/Home Assistant 采集
2. 采集程序把数据 `POST` 到 `/api/env/push`
3. 本项目更新环境状态

如果你现场已经有米家 App 联动，也建议额外保留一条本地数据链路，不要只依赖米家云。

## 为什么分开接

### 温湿度传感器

这类设备通常上报的是标准化传感器数据，例如：

- `temperature`
- `humidity`
- `pressure`
- `battery`
- `linkquality`

最适合本项目作为“环境监测源”读取。

### 空调插座二代 / 空调伴侣二代

这类设备本质更像：

- 带红外能力的空调控制器
- 或带功率统计/插座控制的网关类设备

它既不是当前项目的 Modbus 传感器，也不适合直接复用 `env_sensors` 逻辑。

建议后续单独挂到 `hvac_devices`，作为空调控制设备来做。

你这次发来的截图已经能确认：

- 设备名：`米家空调伴侣2`
- 已接入 Wi-Fi
- 局域网 IP：`192.168.60.114`

但它不能直接当蓝牙温湿度网关来用。

## 已增加的项目能力

本次已新增：

- `services/mqtt_env_bridge.py`
- `services/miio_hvac.py`
- `env_update_loop()` 支持 `source_type = "mqtt"`
- `env_update_loop()` 支持 `source_type = "push"`
- `hvac_update_loop()` 支持 `protocol = "miio"`
- `/api/env/debug?device_id=...` 支持查看 MQTT 调试信息
- `/api/env/push` 支持外部推送环境数据
- `/api/hvac/debug?device_id=...` 支持查看空调伴侣直连状态

## 配置示例

把 `config.json` 里的 `env_sensors` 增加一条类似配置：

```json
{
  "id": "env_xiaomi_room_1",
  "name": "小米温湿度传感器",
  "source_type": "mqtt",
  "features": {
    "temperature": true,
    "humidity": true,
    "illuminance": false,
    "noise": false,
    "pm25": false,
    "pm10": false,
    "pressure": true
  },
  "mqtt": {
    "host": "127.0.0.1",
    "port": 1883,
    "topic": "zigbee2mqtt/xiaomi_th_01",
    "availability_topic": "zigbee2mqtt/bridge/state",
    "qos": 0,
    "stale_after_sec": 7200,
    "field_map": {
      "temp": "temperature",
      "hum": "humidity",
      "pressure": "pressure"
    }
  }
}
```

## Zigbee2MQTT 常见消息示例

```json
{
  "battery": 100,
  "humidity": 46.2,
  "linkquality": 87,
  "temperature": 24.8,
  "pressure": 1008.4
}
```

项目会自动映射到：

- `temp`
- `hum`
- `pressure`

如果你的 topic 字段名不同，可以改 `field_map`。

## 蓝牙 push 配置示例

```json
{
  "id": "env_ble_room_1",
  "name": "蓝牙温湿度传感器",
  "source_type": "push",
  "features": {
    "temperature": true,
    "humidity": true,
    "pressure": false
  },
  "push": {
    "stale_after_sec": 1200
  }
}
```

外部采集程序推送示例：

```http
POST /api/env/push
Content-Type: application/json

{
  "device_id": "env_ble_room_1",
  "temp": 24.6,
  "hum": 48.2,
  "battery": 92,
  "rssi": -67
}
```

## 调试方法

启动项目后查看：

- `GET /api/env/status`
- `GET /api/env/debug?device_id=env_xiaomi_room_1`

`debug` 接口会返回：

- MQTT broker 连接状态
- 订阅 topic
- 最近一条 payload
- 最后更新时间
- 当前解析后的状态

## 关于空调伴侣二代

如果你这次还想把空调开关、模式、温度一起接进系统，建议下一步这样做：

1. 先拿到 `token`
2. 用本项目 `hvac_devices` 的 `protocol = "miio"` 直连测试
3. 如果确认本地协议不可用，再让 Home Assistant 做中间桥

不建议把空调伴侣二代混进 `env_sensors`，否则后面控制逻辑会很乱。

### 空调伴侣配置示例

```json
{
  "id": "hvac_miio_1",
  "name": "机房空调1.5挂机",
  "brand": "Xiaomi",
  "model": "KTBL02LM",
  "protocol": "miio",
  "ip": "192.168.60.114",
  "token": "在此填写32位token",
  "poll_interval_ms": 5000,
  "visible": true,
  "enabled": true
}
```

可用接口：

- `GET /api/hvac/status`
- `GET /api/hvac/debug?device_id=hvac_miio_1`
- `POST /api/hvac/control`
