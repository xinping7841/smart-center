# AI Code Markers

Last updated: 2026-05-22

这份文档定义代码中的本地模型理解标记。目标不是把每一行代码都解释一遍，而是让本地模型和后续开发者能快速判断“这个文件负责什么、不能负责什么、修改风险在哪里、应该从哪里继续查”。

## 标记格式

在 Python、JavaScript、CSS、HTML 中都使用普通注释，不使用特殊语法，方便 `rg`、编辑器和本地模型统一读取。

```text
AI_MODULE: 模块唯一名称。
AI_PURPOSE: 这个文件或模块的主要职责。
AI_BOUNDARY: 这个文件不应该承担的职责，避免后续继续变成大杂烩。
AI_DATA_FLOW: 主要数据来源、缓存、输出对象或前端渲染目标。
AI_RUNTIME: 运行方式、定时轮询、服务入口或被谁调用。
AI_RISK: 修改风险，尤其是物理设备控制、授权、外部接口兼容。
AI_COMPAT: 不能随意删除或改名的路由、字段、协议、前端全局函数。
AI_SEARCH_KEYWORDS: 给本地模型和人类搜索用的关键词。
```

## 风险等级

- `高`: 直接影响强电、时序电源、投影、空调、关机、唤醒、授权、生产启动。
- `中`: 影响页面加载、状态显示、缓存、轮询、接口字段兼容。
- `低`: 只影响纯展示、文档、工具脚本，且不触发真实设备动作。

## 使用规则

- 新模块必须先写模块头，再写功能。
- 老模块拆分时，先在旧文件写清边界，再把实现迁移到 `services/`、`runtime/`、`static/js/views/` 或未来 `modules/`。
- 对真实设备控制路径，必须写清楚锁、延迟确认、缓存刷新和外部兼容字段。
- 不要写“把 value 赋给 x”这类无效注释，模型会学坏，人也会看烦。

## 本地模型推荐读取顺序

1. `docs/AI_NAVIGATION.md`
2. `docs/MODULE_INDEX.yaml`
3. 当前文件顶部的 `AI_*` 标记
4. 当前文件中的 `AI map:`、`Safety:`、`Compatibility:` 注释
5. 相关 API 或 service/core 文件
